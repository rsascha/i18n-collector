# Feature: i18n End-to-End-Strecke (Web-UI ↔ API)

## Kontext
Anschluss an [2026-05-21-spring-api-eval.md](2026-05-21-spring-api-eval.md).
Die Web-UI (`projects/web-ui`) soll Übersetzungen über die Spring-API (`projects/api`) laden und fehlende Keys an die API zurückmelden. Ziel dieser Iteration: erste lauffähige Strecke, in der wir live sehen können, wie i18next mit dem Backend redet — insbesondere wie der `missingKey`-Request aussieht.

## Scope dieser Iteration

### `projects/api` — zwei neue Routen

1. **Übersetzungen ausliefern**
   - Wird von `i18next-http-backend` als `loadPath` aufgerufen.
   - i18next ersetzt darin `{{lng}}` und `{{ns}}`. Vorschlag: `GET /i18n/{lng}/{ns}` — der aktuelle Wert in `i18n.ts` (`http://localhost:8080/i18n`) hat die Platzhalter noch nicht und muss entsprechend angepasst werden.
   - Response: JSON-Objekt `{ key: value, ... }` aus der `translations`-Tabelle für die gegebene Locale (Namespace zunächst ignoriert bzw. fest „common").
   - Kein Auto-Übersetzen bei fehlenden Keys in diesem Schritt — Bedrock-Calls bleiben aus.
   - CORS für die Web-UI (`http://localhost:3000`) freigeben.

2. **Missing-Key-Handler**
   - Wird von `i18next-http-backend` als `addPath` aufgerufen, sobald `saveMissing: true` aktiv ist.
   - Vorschlag: `POST /i18n/{lng}/{ns}`.
   - Body-Format hier noch offen — genau das wollen wir per Console-Log erst einmal beobachten.
   - **Verhalten in diesem Schritt: nur Console-Log** (Locale, Namespace, Body, ggf. relevante Header). Kein DB-Insert, keine Übersetzung.
   - Antwort: `200 OK`, Body leer.

### `projects/web-ui` — i18n aktivieren

> ⚠️ Per `projects/web-ui/AGENTS.md`: Diese Next.js-Version hat Breaking Changes. Vor jeder Änderung in `src/app/` oder am i18n-Setup die relevanten Guides unter `node_modules/next/dist/docs/` lesen.

1. **`src/i18n/i18n.ts`**
   - Aktuell unvollständig: `options` wird definiert, aber nicht an `i18n.init(...)` übergeben — das nachholen.
   - `i18next-http-backend` als Plugin registrieren.
   - `backend.loadPath` und `backend.addPath` auf die neuen API-Routen setzen.
   - `saveMissing: true`, damit `addPath` überhaupt feuert.
   - `missingKeyHandler` darf zusätzlich loggen, das eigentliche Reporten übernimmt aber das Backend-Plugin.
   - Defaults: `fallbackLng: "en"`, `supportedLngs: ["en", "de"]`, `ns: ["common"]`, `defaultNS: "common"`.
   - Client-only Init — die konkrete Einbindung (Provider in Layout / Hook / etc.) richtet sich nach den Next-Docs in `node_modules/next/dist/docs/`.

2. **`src/app/page.tsx`**
   - Statt hartkodiertem `<p>Hallo</p>` einen übersetzten String rendern, z. B. Key `greeting` → "Hallo Welt" / "Hello World".
   - Sprache: vorerst aus `navigator.language` ableiten reicht. **Kein** Sprachumschalter in diesem Schritt.

## Abnahme
1. `pnpm dev` im `projects/web-ui` startet, keine i18n-bezogenen Fehler in der Browser-Console.
2. `page.tsx` zeigt einen tatsächlich übersetzten String (setzt voraus, dass mindestens ein passender Seed-Eintrag in `translations` existiert — Seed-Strategie ggf. Teil der Folge-Iteration).
3. Ein **nicht** existierender Key löst einen Request gegen `POST /i18n/{lng}/{ns}` aus und erscheint im API-Console-Log. Den geloggten Request-Inhalt schauen wir uns gemeinsam an, bevor das Persistieren entworfen wird.

## Was NICHT in diesem Schritt
- Kein DB-Schreiben aus dem missing-key-Endpoint.
- Kein Bedrock-Call im API.
- Kein Sprachumschalter im UI.
- Keine Tests (außer manuelle Abnahme oben).
- Kein Auto-Seed der `translations`-Tabelle — falls die Tabelle leer ist, ist das in Ordnung, dann sehen wir nur missing-key-Logs.

## Offene Punkte / vor Implementierung klären
1. **Route-Shape**: `/i18n/{lng}/{ns}` vs. Query-Params (`/i18n?lng=…&ns=…`) — i18next-http-backend kann beides per `loadPath`-Template. Empfehlung: Pfad-Variante, weil sie ohne extra Konfiguration auch in der Browser-DevTools-URL lesbar ist.
2. **Namespace**: festes `common` reicht für jetzt, oder soll `ns` schon in die DB (`translations`-Tabelle hat aktuell nur `message_key` + `locale`)? Empfehlung: ignorieren in dieser Iteration, später ergänzen.
3. **Response-Format des missing-key-Endpoints**: i18next-http-backend erwartet kein bestimmtes Format — `200` ohne Body ist sicher.
4. **CORS-Konfiguration**: global per `@CrossOrigin` am Controller oder via `WebMvcConfigurer`? Empfehlung: `WebMvcConfigurer`, damit beide Routen zentral abgedeckt sind.

### Abnahme — bestätigt ✅
Alle Abnahmekriterien manuell durchgespielt:
1. `pnpm dev` startet ohne i18n-Fehler.
2. `page.tsx` rendert den übersetzten Seed-String.
3. `t("unknown.key")` löst einen `POST /i18n/{lng}/{ns}` aus, der im API-Log erscheint.

### Beobachtung: Body-Format des missing-key-POST
Roh-Log-Zeile von der API:

```
Missing-key report: lng=en, ns=common, body={unknown.key=unknown.key}
```

Daraus abgeleitet:
- **Body ist ein einfaches JSON-Objekt** `{ <key>: <defaultValue> }` — Spring deserialisiert das in eine `Map<String, String>` (deshalb die `=` in der `toString()`-Ausgabe).
- **Mehrere Missing-Keys werden gebündelt**: i18next-http-backend wird beim parallelen Auftreten mehrerer Misses mehrere Einträge im selben Map-Objekt senden (zu verifizieren, sobald wir tatsächlich mehrere Keys gleichzeitig haben).
- **Wert = `defaultValue`**: ohne expliziten `defaultValue` setzt i18next den Key-Namen selbst als Wert. Für eine Bedrock-Übersetzung ist das untauglich — wir brauchen entweder einen sinnvollen Default vom Aufrufer (`t("greeting", "Hello World")`) oder einen Source-Locale-Fallback im API (z.B. „nimm immer den `en`-Eintrag als Übersetzungs-Input").
- **`ns` und `lng` kommen aus der URL**, nicht aus dem Body — gut, weil der Body sonst noch wachsen müsste.

→ Damit ist das Format klar genug, um in der nächsten Iteration ein konkretes DTO (`Map<String, String>`) und die Persistierungs-/Bedrock-Strategie zu modellieren.

## Folge-Iteration: Server-Side i18n-Proxy (Stand 2026-05-21)

### Entscheidungen aus dem zweiten Ask-Before-Development-Lauf
- **Architektur**: Option (a) — Next.js Route Handler als dünner Proxy. react-i18next bleibt im Client; nur die Backend-URL zeigt jetzt auf `/api/i18n/...` (same-origin) statt direkt auf `http://localhost:8080`.
- **`t(key, defaultValue)`-Konvention**: jeder `t()`-Aufruf liefert einen englischen `defaultValue` mit. Erreicht (1) Flicker-freien Initial-Render, (2) sinnvollen englischen Source-Text im missing-key-Body, (3) graceful Failure-Behavior, wenn der API-Call hängt. **Reine Konvention** — im Doc festgehalten, kein Tooling-Zwang.
- **API-Base-URL**: aus Env-Var `API_BASE_URL` (Default `http://localhost:8080`) im Route Handler — server-side, nicht im Client-Bundle.
- **CORS**: `TranslationWebConfig.java` ersatzlos gelöscht. Der Browser redet nicht mehr direkt mit der Spring-API.
- **Caching**: keins in dieser Iteration — Route Handler nutzt `cache: 'no-store'`. Änderungen in der DB schlagen sofort durch.

### Neu / Geändert

#### `projects/web-ui`
- `src/app/api/i18n/[lng]/[ns]/route.ts` (neu) — Next.js Route Handler mit `GET` und `POST`. Beide proxyen 1:1 an `${API_BASE_URL}/i18n/{lng}/{ns}`. Nutzt den globalen `RouteContext`-Helper für typsichere `params`. `params` ist Promise (Next 15+), wird `await`et. GET reicht den Upstream-Body inkl. `content-type` durch; POST reicht den Request-Body durch und antwortet mit dem Upstream-Status, leerem Body.
- `src/i18n/i18n.ts` — `loadPath`/`addPath` auf relative URL `/api/i18n/{{lng}}/{{ns}}` umgestellt; `API_BASE`-Konstante entfernt.
- `src/app/page.tsx` — `t("greeting", "Hello World")` und `t("unknown.key", "Unknown Key Test")` als Demonstration der defaultValue-Konvention.
- `.env.example` (neu) — `API_BASE_URL=http://localhost:8080` mit erklärendem Kommentar.
- `.gitignore` — `!.env.example`-Exception ergänzt, damit das Template committed werden kann.

#### `projects/api`
- `src/main/java/de/actyvyst/api/translation/TranslationWebConfig.java` (gelöscht) — CORS-Mapping nicht mehr nötig.

### Build-Status
- `pnpm exec tsc --noEmit` → grün.
- `pnpm lint` → grün.
- `./mvnw compile` → grün.

### Designentscheidungen
- **Body-Pass-Through im POST per `await req.text()`**: kein JSON-Parsing im Proxy, der Body wird byte-identisch weitergereicht. Spring-Controller bekommt also exakt dasselbe wie zuvor — keine Format-Verfälschung.
- **`new Response(upstream.body, …)` im GET**: streamt den Upstream-Body, kein Buffering. Bei kleinen Translation-Maps egal, bei sehr großen späteren Payloads relevant.
- **`encodeURIComponent` für `lng`/`ns`**: defensive Verteidigung gegen URL-Injection, auch wenn die Werte im aktuellen Setup nur aus Path-Params kommen.
- **Default-Fallback `"http://localhost:8080"` im Code**: bewusst nicht failen, wenn die Env-Var fehlt — Dev-Setup soll out-of-the-box laufen. Für Prod-Deploy muss `API_BASE_URL` explizit gesetzt werden.

### Konvention (verbindlich für weitere UI-Arbeit)
- **`t(key, defaultValue)` immer mit zweitem Argument aufrufen.** Der `defaultValue` ist der englische Source-Text und gleichzeitig der Bedrock-Input für die spätere Übersetzung. Beispiele: `t("greeting", "Hello World")`, `t("button.save", "Save")`.

### Abnahme — bestätigt ✅
End-to-End manuell durchgespielt. Echte Log-Zeile aus dem laufenden System:

```
2026-05-21T21:54:29.214+02:00  INFO 37925 --- [api] [nio-8080-exec-6] d.a.a.translation.TranslationController  : Missing-key report: lng=en, ns=common, body={unknown.key=Unknown Key Test}
```

Daraus ablesbar:
- **Der Next-Proxy funktioniert**: der `POST` kommt im Spring-Controller auf `:8080` an, also wurde er korrekt vom Route Handler weitergereicht. Im Browser-Network-Tab geht der Request gegen `/api/i18n/en/common` (same-origin), kein direkter `:8080`-Aufruf mehr.
- **Die `t(key, defaultValue)`-Konvention wirkt**: der Body enthält `unknown.key=Unknown Key Test` statt `unknown.key=unknown.key` wie noch in der ersten Iteration. Damit ist der englische Source-Text Teil des missing-key-Reports und kann direkt als Bedrock-Input dienen, sobald wir die Persistierung bauen.
- **CORS ist nicht mehr nötig** — der Controller antwortet sauber, obwohl `TranslationWebConfig` gelöscht wurde, weil alle Browser-Requests gegen `localhost:3000` gehen und der Next-Server server-side gegen `:8080` redet.

## Folge-Iteration: Missing-Key-Persistierung (Stand 2026-05-21)

### Entscheidungen aus dem dritten Ask-Before-Development-Lauf
- **Status-Modell**: neue `TranslationSource`-Variante `PENDING`. Lifecycle: `PENDING` (vom missing-key-POST) → `AI` (nach Bedrock-Übersetzung in der nächsten Iteration). Eine zusätzliche `status`-Spalte wäre Overkill, NULL-`value` als Marker fragiler.
- **Welche Zeilen werden geschrieben**: nur die Ziel-Locale aus der URL — nicht spekulativ alle `supportedLngs`, nicht zusätzlich `en` als Beigabe.
- **`en`-Sonderfall**: wenn `lng == "en"`, dann `source = MANUAL` und `value = defaultValue`. Begründung: der englische `defaultValue` aus dem UI IST der kanonische englische Text — Bedrock darf eine `en`-Zeile nie anfassen.
- **`value` bei PENDING**: defaultValue als Platzhalter. Schema-Constraint `value NOT NULL` aus V1 bleibt intakt. Übergangs-Verhalten ist identisch zum aktuellen UI (zeigt sowieso den `defaultValue`, bis i18next die Locale-Daten geladen hat).
- **Conflict-Behavior**: `ON CONFLICT (message_key, locale) DO NOTHING`. Idempotent, schützt bestehende `AI`-Übersetzungen vor versehentlichem Überschreiben.
- **Sync/Async**: synchron im selben POST. i18next-http-backend ist clientseitig fire-and-forget, ein paar Millisekunden DB-Latenz spielen keine Rolle. Async-Konfiguration kommt erst mit Bedrock.

### Neu / Geändert

#### `projects/api`
- `src/main/java/de/actyvyst/api/translation/TranslationSource.java` — Enum-Wert `PENDING` ergänzt.
- `src/main/resources/db/migration/V3__add_pending_source.sql` (neu) — droppt den V1-CHECK-Constraint defensiv (`IF EXISTS`) und legt ihn mit `('MANUAL','AI','PENDING')` neu an.
- `src/main/java/de/actyvyst/api/translation/TranslationRepository.java` — native `@Modifying @Query`-Methode `insertIfAbsent(messageKey, locale, value, source)` mit `INSERT … ON CONFLICT (message_key, locale) DO NOTHING`. `created_at`/`updated_at` per `now()` direkt aus SQL (umgeht die JPA-Lifecycle-Hooks, weil wir ja keinen Entity-Round-Trip machen).
- `src/main/java/de/actyvyst/api/translation/TranslationService.java` (neu) — kapselt die Entscheidung `lng == "en"` → `MANUAL` / sonst `PENDING` und ruft das Repository pro Map-Eintrag auf. `@Transactional`, damit alle Inserts eines POST in einer Tx laufen.
- `src/main/java/de/actyvyst/api/translation/TranslationController.java` — Body-Typ von `Object` auf `Map<String, String>` verschärft (das beobachtete Format passt jetzt). POST ruft `translationService.recordMissingKeys(...)`, Log-Zeile bleibt für die Beobachtung.

### Build-Status
- `./mvnw compile` → grün.

### Designentscheidungen
- **Native SQL statt JPA-`save`**: `ON CONFLICT DO NOTHING` ist Postgres-spezifisch und in JPA nicht direkt ausdrückbar. Alternativen (try/catch auf `DataIntegrityViolationException`, `findByKeyAndLocale` davor) sind entweder schmutziger oder rennen in Race-Conditions. Native Query ist die saubere Variante; wir haben uns ohnehin schon im V1-Schema auf Postgres festgelegt.
- **`created_at`/`updated_at` direkt aus SQL (`now()`)**: bei nativem Insert greifen die `@PrePersist`-Hooks der Entity nicht. Die DB-Defaults aus V1 würden zwar reichen, aber explizit `now()` im SQL macht das deutlich, falls die Spalten-Defaults später wegfallen.
- **`SOURCE_LOCALE = "en"` als Konstante im Service**: bewusst nicht in `application.yml`, weil das aktuell eine semantische Annahme über das System ist (englischer `defaultValue` aus UI). Wenn das später konfigurierbar werden muss, ziehen wir es nach.
- **`body == null`/leer-Check im Controller**: defensive Belt-and-suspenders; `@RequestBody(required = false)` lässt theoretisch leere Bodies durch, dann hätten wir eine NullPointer-Falle.

### Offene Punkte (für die Bedrock-Iteration)
1. **Bedrock-Trigger**: was stößt die Übersetzung an? Optionen: (a) sofort am Ende des POST in der Service-Methode (synchron — Latenz steigt), (b) `@Async` direkt im selben Service, (c) Cron/Scheduler, der `PENDING`-Zeilen pollt, (d) Outbox/Queue. Klärung in der nächsten Iteration.
2. **Was ist der Bedrock-Input**: der `defaultValue` aus der PENDING-Zeile selbst (steht ja in `value`) — oder explizit die `en`-Zeile lookup, falls vorhanden? Wenn man die en-Zeile als kanonisch ansieht, wäre das robuster gegen unterschiedliche defaults aus verschiedenen UI-Stellen.
3. **`updated_at` nach Bedrock**: Update muss `updated_at = now()` und `source = AI` setzen — vermutlich auch über ein natives Statement, oder per JPA-Entity-Update.

### Manuelle Abnahme (zu fahren)
1. API mit V3-Migration starten — Flyway sollte `V3` automatisch nachziehen.
2. Bestehende `unknown.key`-Zeilen aus der DB löschen, falls sie aus Vortests in die `de`-Locale gespült wurden (sonst greift `DO NOTHING`).
3. UI laufen lassen mit Browser-Sprache `de` (oder `en`).
4. `SELECT * FROM translations WHERE message_key = 'unknown.key';` — erwartet:
   - Bei `lng=de`: Zeile `de/unknown.key=Unknown Key Test, source=PENDING`.
   - Bei `lng=en`: Zeile `en/unknown.key=Unknown Key Test, source=MANUAL`.
5. Mehrfaches Reload: nur die erste Zeile bleibt, keine Duplikate.

#### Teil-Abnahme — Stand 2026-05-21 Abend
- ✅ `en`-Pfad bestätigt durch DB-Query:

  ```
   id | message_key | locale |      value       | source
  ----+-------------+--------+------------------+--------
    4 | greeting    | en     | Hello World      | MANUAL
    5 | unknown.key | en     | Unknown Key Test | MANUAL
  ```
- ❌ `de`/`PENDING`-Pfad **noch nicht verifiziert**, weil der Sprachumschalter im UI (siehe nächste Sektion) nicht zuverlässig zwischen `en` und `de` umschaltet. Workaround für morgen: entweder Bug fixen ODER per `curl -X POST http://localhost:3000/api/i18n/de/common -H 'Content-Type: application/json' -d '{"unknown.key":"Unknown Key Test"}'` direkt triggern und DB nachprüfen.

### Diagnose-Befunde (das eigentliche Ergebnis)

Was wie ein React/i18next-Re-Render-Bug aussah, waren **zwei zusammenwirkende Ursachen**, beide nicht im UI-Code:

1. **`translations`-Tabelle hatte keine DE-Zeile.** Spring-Endpoint `GET /i18n/de/common` lieferte `{}`. Die V2-Migration sieht `('greeting','de','Hallo Welt')` zwar vor, war in der laufenden Dev-DB aber nicht (mehr) vorhanden — vermutlich manuell entfernt während früherer PENDING-Tests. Konsequenz: i18next-http-backend lud ein leeres Bundle für DE; `t(key, { lng: "de" })` fiel via `fallbackLng` korrekt auf EN zurück → **Texte wechselten nie**.

2. **`i18n.resolvedLanguage` ist nicht die ausgewählte Sprache, sondern die effektiv aufgelöste.** Wenn DE leer ist und EN als Fallback greift, bleibt `resolvedLanguage="en"` selbst nach `i18n.changeLanguage("de")`. Unser `useSyncExternalStore`-Snapshot las diesen Wert → blieb konstant → React rendert nicht neu → der Eindruck „Sprache hinterher" entsteht. **Snapshot muss `i18n.language` lesen** (die formal gewählte Sprache, unabhängig vom Fallback).

Vollständige Event-Sequenz aus `diagnose-i18n-events.spec.ts` vor dem Fix:
```
[diag] before-listeners      language=en  resolvedLanguage=en  hasEN=true  hasDE=false  t(greeting)=Hello World
[diag] event:languageChanged(de)  language=de  resolvedLanguage=en  hasEN=true  hasDE=true  t(greeting)=Hello World
[diag] event:loaded({"de":{"common":true}})  language=de  resolvedLanguage=en  hasEN=true  hasDE=true  t(greeting)=Hello World
[diag] AFTER-2s              language=de  resolvedLanguage=en  hasEN=true  hasDE=true  t(greeting,lng=de)=Hello World
```
Schlüsselzeile: `hasDE=true` aber DE-Bundle ist leer ⇒ `t(...,{lng:"de"})` liefert EN-Wert. Damit war klar, dass das Problem _vor_ React liegt.

### Designentscheidungen
- **`window.__i18n` nur in Dev**: `process.env.NODE_ENV !== "production"`-Guard wird von Next zur Build-Zeit aufgelöst und durch tree-shaking eliminiert. Diagnose-Spec funktioniert lokal, Prod-Bundle ist frei davon.
- **Diagnose-Spec behalten statt löschen**: i18next-Diagnose ist nicht-trivial, das Setup (Listener installieren via `page.evaluate`, Console-Capture via `page.on("console")`) ist wiederverwendbar. Kommentar-Header weist drauf hin, dass das Skript ad-hoc läuft und nicht Teil der CI-Suite ist.
- **V4 statt direkter SQL-Edit**: reproduzierbar für andere Devs, dokumentiert das ursprüngliche Setup-Intent. `ON CONFLICT DO NOTHING` schützt davor, manuelle Edits in der Dev-DB zu überschreiben.
- **`exact: true` im Selector**: gefangen durch ersten Test-Lauf — `getByRole("button", { name: "EN" })` matcht per Default Substrings im accessible name. Lehre: bei kurzen Button-Labels (`EN`/`DE`) immer `exact: true`.

### Designentscheidungen (Nachtrag)
- **Modell-Wahl `claude-haiku-4-5`**: ~1s Latenz, niedrige Kosten, für 2-5-Wort-UI-Strings überreichlich akkurat. Sonnet wäre Overhead.
- **AWS-Credential-Export im API-Makefile statt globalem Shell-Wrapper**: bleibt im API-Projekt, andere Subsysteme (web-ui, e2e-tests) sehen keine AWS-Side-Effects. Identifizierbar an der `@if command -v aws`-Klausel.

### Offene Punkte für die nächste Iteration
1. **Timezone-Inkonsistenz in `created_at`/`updated_at`** — id=7 hat `createdAt: 2026-05-22T12:55:02Z` (Native-SQL `now()` in V3-`insertIfAbsent` → DB-Lokalzeit als „UTC" ausgegeben) und `updatedAt: 2026-05-22T10:58:31Z` (echter UTC via `@PreUpdate`/`Instant.now()`). 2h-Diff durch CEST. Fix-Optionen: (a) Repository-SQL auf `now() AT TIME ZONE 'UTC'`, (b) Spalten via Migration auf `TIMESTAMP WITH TIME ZONE` heben, (c) JPA-Hook auch für native Inserts triggern (geht nicht ohne Round-Trip). Empfehlung: (a) — minimaler Eingriff.
2. **Re-Translate für AI-Zeilen** — wenn man eine bestehende AI-Übersetzung verwerfen + neu übersetzen will, fehlt der Flow. Optionen: separater „Reset to PENDING"-Button, oder „Force re-translate" mit Bestätigungs-Dialog.
3. **Batch-Translate** — bei vielen PENDING-Zeilen wäre ein „alle übersetzen"-Button nützlich. Aktuell pro-Zeile.
4. **Fehlerbild bei Bedrock-Fehlern** — aktuell wird die Spring-Fehlermeldung im Inline-Error gezeigt; bei z. B. „model access denied" wäre eine freundlichere UI-Meldung mit Setup-Link sinnvoll. Auch sinnvoll: Stack-Traces in der Spring-Response unterdrücken (z. B. `server.error.include-stacktrace=never`), damit der Client nicht das ganze JVM-Innenleben sieht.
5. **E2E-Test für `/i18n`** — die Seite ist noch nicht in Playwright abgedeckt. Sinnvoller Folge-Test, sobald Bedrock-Credentials in einer Test-Umgebung verfügbar sind (oder gemockt werden müssen).
6. **AWS-Credential-Expiry-UX** — SSO-Tokens laufen typischerweise nach 1h ab. Aktuell merkt der User es erst beim nächsten Translate-Click (500-Fehler). Sinnvoll: präventiver Health-Check oder klarere Fehlermeldung „SSO-Session abgelaufen, `aws sso login` ausführen".
